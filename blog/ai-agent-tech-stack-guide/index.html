AI Agent 技术栈全景：从核心架构到必备框架

2025-2026 年是 AI Agent 从概念走向工程落地的关键时期。本文将从核心架构、设计模式、主流框架、协议层和生产实践五个维度，系统梳理构建 AI Agent 所需的完整技术栈。

一、什么是 AI Agent

AI Agent 不是简单的 "LLM + Prompt"。它是一个能够感知环境、自主规划、使用工具、迭代执行的智能系统。与传统的单轮问答不同，Agent 具备：

  • 自主性 — 能自行决定下一步动作，而不是等待用户指令
  • 工具使用 — 能调用外部 API、执行代码、操作数据库
  • 记忆 — 能记住上下文、历史交互和学到的知识
  • 规划 — 能将复杂任务分解为子步骤并逐步执行
  • 反思 — 能评估自身输出质量并自我纠正

二、核心架构

2.1 LLM（大脑）

LLM 是 Agent 的推理引擎，负责理解用户意图、制定计划、选择工具、生成回复。主流选择：

模型厂商特点适用场景
GPT-4o / o3OpenAI综合最强，工具调用稳定通用 Agent、复杂推理
Claude 4 Opus/SonnetAnthropic长上下文、代码能力强代码 Agent、文档分析
Gemini 2.5 ProGoogle多模态、长上下文多模态 Agent
DeepSeek-V3DeepSeek性价比极高成本敏感场景
Llama 3.3 / 4Meta开源可私有化部署数据隐私要求高
Qwen 3阿里中文能力强中文场景

2.2 Memory（记忆）

Agent 的记忆分为三层：

  • 短期记忆 (Short-term) — 当前对话上下文，受 context window 限制。实现方式：消息列表 + 摘要压缩。
  • 长期记忆 (Long-term) — 持久化的知识库，通常通过 RAG（检索增强生成）实现。向量数据库选型：Pinecone、ChromaDB、pgvector、Milvus。
  • 情景记忆 (Episodic) — 过去任务的执行经验和反馈。代表项目：Mem0。

2.3 Tools（工具）

工具是 Agent 与外部世界交互的接口。主要通过两种机制实现：

  • Function Calling — OpenAI/Claude 原生支持，LLM 输出结构化的工具调用请求
  • MCP (Model Context Protocol) — Anthropic 提出的开放协议，将工具抽象为独立服务。详见下文 MCP 章节。

常见工具类型：代码执行、Web 搜索、数据库查询、API 调用、文件读写、浏览器操作。

2.4 Planning（规划）

Agent 的规划能力决定了它处理复杂任务的上限。核心方法：

  • ReAct — Reasoning + Acting 交替执行，每步先思考再行动
  • Chain-of-Thought (CoT) — 逐步推理，提高准确性
  • Task Decomposition — 将大任务拆分为可执行的子任务
  • Plan-and-Execute — 先制定完整计划，再逐步执行

三、Agent 设计模式

3.1 ReAct Loop

最基础也最常用的模式。Agent 在 Think → Act → Observe 之间循环，直到任务完成。

Thought: I need to find the current stock price of AAPL.
Action: search("AAPL stock price today")
Observation: AAPL is trading at $198.50.
Thought: Now I have the price. Let me calculate the portfolio value.
Action: calculate(shares=100, price=198.50)
Observation: Portfolio value = $19,850.00
Thought: I have all the information. Let me respond.

3.2 Multi-Agent Collaboration

将复杂任务分配给多个专门化的 Agent：Planner 负责拆解、Coder 负责编码、Reviewer 负责审查、Executor 负责执行。

代表框架：CrewAI、AutoGen。

3.3 Tool-Augmented Agent

LLM 根据当前状态选择最合适的工具调用。通过 Function Calling 或 MCP 实现工具发现和调用。

代表框架：PydanticAI、OpenAI Agents SDK。

3.4 Graph-Based Workflow

将 Agent 逻辑建模为有向图，每个节点是一个处理步骤，边可以是条件分支。支持循环、状态持久化、Human-in-the-loop。

代表框架：LangGraph。

3.5 Agent Handoff

Triage Agent 负责识别用户意图，然后无缝将对话委派给专门的 Agent（Sales/Support/Tech），同时保持上下文完整传递。

代表框架：OpenAI Agents SDK (Handoff 机制)。

四、主流框架全景对比

4.1 LangGraph — 状态图编排

LangGraph 是 LangChain 团队推出的 Agent 编排框架，核心理念是将 Agent 工作流建模为有状态的有向图。

核心特性：

  • Cycles & Branching — 支持循环和条件分支，不局限于 DAG
  • Persistence — 内置状态检查点，支持暂停/恢复和 time-travel 调试
  • Human-in-the-loop — 节点间可插入人工审批步骤
  • Streaming — 每个节点实时流式输出
from langgraph.graph import StateGraph, START, END

graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tools)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_continue, {
    "continue": "tools",
    "end": END
})
graph.add_edge("tools", "agent")  # loop back

app = graph.compile()

适用场景：需要复杂控制流、状态管理、Human-in-the-loop 的生产级 Agent。

参考链接： LangGraph 文档 · GitHub

4.2 OpenAI Agents SDK — 官方 Agent 框架

OpenAI Agents SDK 是 OpenAI 官方的 Python Agent 框架，设计极简但功能完整。

核心特性：

  • Agent Loop — 内置 ReAct 循环，自动处理 tool 调用和结果回注
  • Handoffs — Agent 之间的无缝委派，保持上下文
  • Guardrails — 输入/输出验证器，防止越界行为
  • Tracing — 内置 OpenTelemetry 追踪
  • MCP 支持 — 原生集成 MCP 协议
from agents import Agent, Runner

agent = Agent(
    name="Research Assistant",
    instructions="You help users research topics.",
    tools=[web_search, file_reader],
    handoffs=[specialist_agent],
)
result = await Runner.run(agent, "Analyze recent AI trends")

适用场景：使用 OpenAI 模型构建生产级 Agent，需要 Handoff 和 Guardrails。

参考链接： 官方文档 · GitHub

4.3 PydanticAI — 类型安全的 Agent 框架

PydanticAI 是 Pydantic 团队打造的 AI Agent 框架，核心设计理念是类型安全和结构化输出。

核心特性：

  • Model-agnostic — 支持 OpenAI、Anthropic、Gemini、Ollama 等
  • Type-safe — 用 Pydantic 模型定义 Agent 输入/输出/依赖
  • Structured Output — 强类型的结构化返回，无需手动 JSON 解析
  • Dependency Injection — 类似 FastAPI 的依赖注入系统
from pydantic_ai import Agent
from pydantic import BaseModel

class CityInfo(BaseModel):
    name: str
    country: str
    population: int

agent = Agent(
    "openai:gpt-4o",
    output_type=CityInfo,  # 强类型输出
    system_prompt="Extract city information."
)
result = await agent.run("Tell me about Tokyo")
print(result.output.population)  # IDE 自动补全 ✓

适用场景：注重类型安全和结构化输出的 Python 项目。

参考链接： 官方文档 · GitHub

4.4 CrewAI — 角色协作 Agent

CrewAI 专注于多 Agent 角色协作，让你定义"团队"而非单个 Agent。

from crewai import Agent, Task, Crew

researcher = Agent(role="Researcher", goal="Find latest AI papers", ...)
writer = Agent(role="Writer", goal="Write blog post", ...)

task1 = Task(description="Research AI agent trends", agent=researcher)
task2 = Task(description="Write a summary article", agent=writer)

crew = Crew(agents=[researcher, writer], tasks=[task1, task2])
result = crew.kickoff()

适用场景：需要多角色协作的业务流程自动化。

参考链接： 官方文档 · GitHub

4.5 AutoGen — 对话式多 Agent

AutoGen (Microsoft) 通过对话驱动多 Agent 协作，Agent 之间通过消息传递协调。

适用场景：需要 Agent 之间自主对话协商的复杂任务。

参考链接： 官方文档 · GitHub

4.6 Google ADK — Agent Development Kit

Google ADK 是 Google 的 Agent 开发套件，紧密集成 Gemini 模型和 Google Cloud 生态。

参考链接： 官方文档 · GitHub

五、协议层：MCP / A2A / AG-UI

5.1 MCP (Model Context Protocol)

MCP 是 Anthropic 提出的开放协议，目标是标准化 LLM 与外部工具/数据源的交互方式。你可以把它理解为 "AI 世界的 USB 接口"。

  • Tools — Agent 可调用的函数（搜索、计算、API 调用等）
  • Resources — Agent 可读取的数据源（文件、数据库等）
  • Prompts — 预定义的 prompt 模板

MCP 的关键价值：一次实现，处处可用。一个 MCP Server 可以被 Claude、ChatGPT、LangGraph、PydanticAI 等任何支持 MCP 的客户端使用。

参考链接： MCP 官方文档 · GitHub 组织

5.2 A2A (Agent-to-Agent Protocol)

A2A 是 Google 提出的 Agent 间通信协议。如果说 MCP 解决的是 "Agent ↔ Tool" 的问题，A2A 解决的是 "Agent ↔ Agent" 的问题。

  • Agent 通过 Agent Card 发布自己的能力
  • 支持跨框架、跨厂商的 Agent 互操作
  • 任务生命周期管理（创建 → 执行 → 完成）

参考链接： A2A 协议规范

5.3 AG-UI (Agent-User Interaction Protocol)

AG-UI 是 CopilotKit 提出的 Agent-to-Frontend 协议，解决 Agent 与 UI 的实时通信。

  • Agent 可以实时更新 UI 状态
  • 支持流式文本、工具调用可视化、状态同步

参考链接： AG-UI 文档

七、生产实践要点

7.1 可观测性

  • Langfuse — 开源 LLM 可观测平台（Tracing、评估、Prompt 管理）
  • LangSmith — LangChain 官方的追踪和评估平台
  • OpenTelemetry — Agent 链路的标准化追踪

7.2 评估 (Evaluation)

  • 正确性 — Agent 是否完成了任务？
  • 效率 — 用了多少步骤、多少 token？
  • 安全性 — 是否遵守了边界约束？
  • 工具： DeepEval、RAGAS、自定义评估套件

7.3 安全与 Guardrails

  • Prompt Injection 防护 — 输入清洗、指令隔离
  • 输出验证 — 结构化校验（Pydantic）、敏感信息过滤
  • 工具权限控制 — 最小权限原则，危险操作需人工确认
  • 工具： Guardrails AI、OpenAI Agents SDK 内置 Guardrails

7.4 部署架构

  • LLM 路由 — LiteLLM 统一 API 代理，支持故障转移和负载均衡
  • 异步执行 — Celery / Ray 处理长时间运行的 Agent 任务
  • 状态管理 — Redis / PostgreSQL 持久化 Agent 状态
  • 容器化 — Docker + Kubernetes 标准化部署

九、总结

AI Agent 技术栈正在快速成熟，核心要素可以归纳为：

  1. LLM 选型 — 根据成本、延迟、能力需求选择合适的模型
  2. 框架选型 — 从 LangGraph / OpenAI Agents / PydanticAI 中选择匹配的编排方式
  3. 协议层 — MCP 已成为工具集成的事实标准，A2A 正在崛起
  4. 记忆系统 — RAG + 向量数据库是长期记忆的标准方案
  5. 生产化 — 可观测性、评估、安全 guardrails 是上线的必备能力

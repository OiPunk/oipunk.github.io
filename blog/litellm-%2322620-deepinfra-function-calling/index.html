Add Missing Function Calling Support for 55 DeepInfra Models in LiteLLM

This post covers my merged fix in LiteLLM, the universal LLM API proxy layer used by thousands of AI applications.

Primary References

  • Pull Request: BerriAI/litellm#22620
  • Files changed: 2 files, +124/-55

The Problem

55 DeepInfra models in LiteLLM's model registry had supports_tool_choice: true and included "tools" in their supported_openai_params, but were missing the supports_function_calling: true flag.

This caused litellm.supports_function_calling() to return False, and upstream frameworks like instructor and langchain would silently skip tool calling for these models entirely.

# Before fix: incorrectly returns False
litellm.supports_function_calling("deepinfra/meta-llama/Llama-3.3-70B-Instruct-Turbo")
# → False (should be True)

Root Cause

The supports_function_calling() function checks a dedicated boolean flag in the model metadata. When this flag is None (not set), it defaults to False — even if the model clearly supports tools via other metadata fields.

The Fix

Added "supports_function_calling": true to all 55 affected DeepInfra model entries in model_prices_and_context_window.json, plus a regression test.

def test_supports_function_calling_deepinfra_llama():
    assert litellm.supports_function_calling(
        model="deepinfra/meta-llama/Llama-3.3-70B-Instruct-Turbo"
    ) is True

Impact

  • 55 models fixed in one patch — broad impact, minimal risk
  • All LiteLLM users calling DeepInfra models now get correct function calling capability detection
  • Upstream frameworks (instructor, langchain) will automatically start using tool calling for these models

Fix Pickle Serialization for All Exception Classes in PydanticAI

This post covers my merged fix in PydanticAI, the official AI agent framework from the Pydantic team.

Primary References

  • Pull Request: pydantic/pydantic-ai#4504
  • Files changed: 2 files, +91/-0

Pickle Round-Trip Overview

The Problem

PydanticAI's custom exception classes (CallDeferred, ApprovalRequired, UnexpectedModelBehavior, ModelAPIError, ModelHTTPError) stored attributes in custom __init__ parameters that were not passed to super().__init__().

This meant pickle.dumps() → pickle.loads() would reconstruct the exception but lose all custom attributes like status_code, model_name, metadata, etc.

In distributed environments (Celery, Ray, multiprocessing), this caused error handlers to receive empty or broken exception objects.

The Fix

Added __reduce__ methods to all 5 exception classes:

class ModelHTTPError(ModelAPIError):
    def __reduce__(self):
        return (
            self.__class__,
            (self.status_code, self.model_name, self.body),
        )

Test Coverage

Added 16 parameterized tests covering every exception class and its variants:

@pytest.mark.parametrize("exc", [
    ModelHTTPError(429, "gpt-4", "rate limited"),
    ModelHTTPError(500, "gpt-4", None),
    ModelAPIError("gpt-4", "connection failed"),
    CallDeferred(metadata={"key": "value"}),
    ApprovalRequired(metadata=None),
    UnexpectedModelBehavior("bad response", body="raw"),
])
def test_pickle_round_trip(exc):
    restored = pickle.loads(pickle.dumps(exc))
    assert type(restored) is type(exc)
    # assert all custom attributes match

Impact

  • 5 exception classes fixed, covering all PydanticAI custom exceptions
  • Enables reliable error propagation in Celery, Ray, multiprocessing workflows
  • PydanticAI is Pydantic's official AI framework — large and growing user base
  • Framework-level infrastructure fix with zero breaking changes
